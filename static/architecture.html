    CHAPTER 01
    为什么需要 Ray？
    客户痛点 · 14 天 vs 4 小时的鸿沟
    第 1 幕 · 售前进场，先听见痛苦
    客户 IT 总监摊开数据：100 万张图，单机 Python + PaddleOCR 串行跑，预估 14 天。 买更猛的单机？GIL 锁住 Python 多线程（↓ 下方图解），加 CPU 也没用；GPU 只有一张，再贵的卡也喂不过来。 客户原话："我们不想成为分布式系统专家，但我们必须 4 小时内出结果。"——这就是 Ray 入场的理由。
    🔒 什么是 GIL？为什么"加 CPU 也没用" 点我展开图解 ▾

    GIL = Global Interpreter Lock（全局解释器锁），是 CPython（官方 Python 实现）里的一把全局大锁。

    核心规则：同一时刻，一个 Python 进程里只能有一个线程在执行 Python 字节码。哪怕你机器有 64 核 CPU，开 64 个 Python 线程跑纯计算，也只有 1 个线程在真正干活，其他 63 个在等锁。

    🔪 厨房比喻：4 个厨师（线程），但厨房只有一把菜刀（GIL）。任何切菜动作都得抢这把刀——再多招厨师也没用，他们只能轮流切。

    💡 所以客户场景里：OCR 是 CPU 密集 + GPU 计算，加再多 CPU 核，Python 多线程也无法并行——只能靠多进程（每个进程一把自己的 GIL）或多机分布式。Ray 走的就是多进程 + 多机路线，绕开 GIL。

    ✅ 绕开方案：多进程（multiprocessing / Ray） ✅ 不受影响：纯 IO、numpy C 扩展（释放 GIL） ❌ 受限：纯 Python CPU 密集计算 📌 PEP 703：CPython 3.13 已实验性支持「无 GIL」，仍未通用
    ⚠️ 单机 Python 的天花板
    Python 是 AI/ML 的绝对主力，但 CPython 的 GIL（见上方图解）限制了真正的多线程并行。客户场景里，单机会撞墙：
    • 100 万张图 OCR，单卡 GPU 串行跑要 14 天
    • 同时还要跑 YOLO 目标检测双模型，单机 GPU 显存爆掉
    • 夜间任务挂了一次，重跑又是 14 天 —— 无容错
    • SSH 脚本 + 自管进程 + 自己写错误恢复 —— 分布式问题甩给开发者
    🚀 Ray 的核心承诺
    "让分布式计算像单机编程一样简单"——加一行 @ray.remote 装饰器，OCR 函数就能在 8 张 GPU 上并发跑。Ray 帮你处理：调度、通信、容错、资源、序列化。
    单机 vs Ray —— 只差一行装饰器（用客户的 OCR 函数演示） 
    # —— 单机版本：14 天 ——
def ocr_one(image_path):
    model = PaddleOCR()
    return model.ocr(image_path)

results = [ocr_one(p) for p in million_images]   # 串行 ❌

# —— Ray 版本：4 小时 ——
@ray.remote(num_gpus=1)
def ocr_one(image_path):
    model = PaddleOCR()
    return model.ocr(image_path)

# 8 张 GPU 并发，Ray 自动调度
refs = [ocr_one.remote(p) for p in million_images]
results = ray.get(refs)
    @ray.remote 这一行装饰器背后，是 GCS / Raylet / Plasma / Worker 四层精心设计的分布式系统在协同运转
    📌 售前一句话 Ray = 用本地 Python 语法表达分布式计算。后面 10 章，就是把 14天→4小时 这个承诺层层兑现。

    CHAPTER 02
    设计哲学推导
    客户问：为什么不是 Spark？为什么是 Ray？
    第 2 幕 · 选型答辩
    客户技术委员会的 Spark 老兵当场质问："我们 Spark 集群现成的，为什么不复用？" 售前架构师不能靠"Ray 更新 / Ray 流行"来糊弄，必须从第一性原理推： OCR + YOLO 是毫秒级有状态推理任务，模型权重要常驻 GPU——这是 Spark 的盲区，也是 Ray 的主场。
    💡 第一性原理
    任何分布式计算框架都要解决三个核心问题：
    1. 调度（Scheduling）：任务该分配给谁执行？
    2. 数据（Data）：任务之间的数据如何传递？
    3. 容错（Fault Tolerance）：机器挂了怎么办？
    三家给出三种答案：
    • Spark：中心化调度 + RDD 血缘 + 重新计算
    • Dask：动态任务图 + 延迟执行 + 单机语义
    • Ray：Actor 模型 + 分布式对象存储 + 自底向上调度
    🎯 关键决策一：为什么 Ray 不用中心化调度？
    中心化调度器（Spark Driver）在批处理够用，但客户的 OCR + YOLO 工作负载有三个独特特征，正好踩中 Spark 三个盲区：
    特征Spark 强项（批处理）客户场景（OCR + YOLO）
    任务粒度粗（分钟级 Stage）细（单图推理 ~50ms）
    任务类型无状态 Map/ReduceOCR/YOLO Actor 常驻显存 + 后处理 Task 混合
    资源需求均匀 CPU异构：8× GPU（推理）+ 16× CPU（后处理）
    🔬 推导
    如果每个毫秒级任务都要经过中心调度器，调度器本身就是吞吐瓶颈。
    Ray 的解法：自底向上（Bottom-Up）调度 —— 先在本地调度，本地不行再找全局。99% 任务在 100μs 内完成本地分配。
    🎯 关键决策二：为什么用共享内存而不是 RPC 传数据？
    传统框架用 RPC（gRPC）在进程间传数据，每次都要序列化 + 反序列化。Ray 选择了 Plasma 共享内存对象存储：
    • 同节点进程间通过 mmap 零拷贝读取，无需序列化
    • 跨节点传输只在边界发生一次
    • 小对象（< 100KB）直接内联在 ObjectRef 中，连共享内存都不碰
    这个设计使 Ray 任务启动达到毫秒级，而 Spark 通常在秒级
    ⚡ 第一性原理 · 直面 Spark 老兵的质问
    客户反问"Spark 集群现成的，为什么不复用？"——售前架构师不能回答"Ray 更新 / Ray 流行"，而要拆出 OCR + YOLO 工作负载的三个本质特征：
    ① 毫秒级任务（一张图 OCR 50ms） · ② 有状态推理（模型权重必须常驻 GPU） · ③ 异构资源（GPU 推理 + CPU 后处理）。
    这三条，每一条都正好踩中 Spark 的结构盲区，又正好是 Ray 的架构主场。下面这张时序动画把它讲清楚：
    ▶ 动画自动播放：上半段是 Spark 的 Stage 启动 + 反复加载模型；下半段是 Ray Actor 一次加载常驻、Task 弹性后处理。这不是"Ray 更新"——是工作负载的本质决定了选型。
    🔬 答辩三问 · 写进 PPT 的版本
    Q1：能不能在 Spark 里跑 OCR？
    能跑，但 Spark 的 Stage 模型决定了：① Executor JVM 启动 + Python 子进程启动 ≈ 秒级；② mapPartitions 里 PaddleOCR 模型每个分区都要重新初始化；③ GPU 资源调度依赖 Spark 3.x 的 RAPIDS 插件，且只支持 NVIDIA stack 的部分算子。跑得起来，跑不快，跑不省。

    Q2：那为什么不"Spark 跑数据，Ray 跑推理"两套混用？
    可以，但需要在 Spark Action（write）和 Ray put 之间反复落盘——跨框架数据来回 = 序列化开销翻倍 + 运维两套集群。客户的 100 万图项目里实测：双框架方案比纯 Ray 慢 2.4×，而且故障定位面要查两个 Web UI。

    Q3：那如果客户已经有 Spark 投资？
    保留 Spark 跑批处理 ETL（夜间日志聚合、报表），新建 Ray 跑 AI 推理——两个集群通过同一份 S3/Lance 数据资产连接，物理隔离、责任清晰。Ray 不是替代 Spark，是补全 Spark 的盲区。
    📌 售前一句话 给客户的答辩稿就三句：调度要快（自底向上，毫秒级）、数据要近（共享内存零拷贝）、容错要简单（无状态组件 + 一个 GCS）。Spark 在三句话上都是另一个答案——所以不是它。

    CHAPTER 03
    核心架构 · 四层分层
    客户集群拓扑：1 Head + 2× GPU 节点 + 4× CPU 节点
    第 3 幕 · 在客户白板上画拓扑
    选型过了，下一个问题立刻砸过来："那机器怎么部署？谁是大脑？谁是手脚？挂了找谁？" 售前架构师拿马克笔在白板上画 4 个组件： GCS（大脑·只在 Head Node）、 Raylet（每节点管家）、 Plasma（每节点数据层）、 Worker（干活的）。 客户的 100 万图任务里，OCR Actor 跑在 GPU 节点的 Worker 里，输入图片在 Plasma，调度由 Raylet 决定，状态由 GCS 记录。
    每个节点都跑 Raylet + Plasma + 多个 Worker；只有 Head Node 跑 GCS（集群唯一中心化组件） · 黄线 = 心跳 + 调度 / 蓝线 = 对象拉取
    🧠 3.1 GCS · 集群的大脑
    GCS（Global Control Service）是 Ray 集群中唯一的中心化组件，运行在 Head Node 上。
    核心设计原则：让其他所有组件无状态。
    数据表内容
    节点表集群中所有节点的存活状态、资源总量
    Actor 表每个 Actor 的位置、状态、重建信息
    对象位置表ObjectRef → 存储节点映射
    函数表@ray.remote 注册的函数定义
    Placement Group 表资源预留组的配置和状态
    Job 表运行中的作业信息
    🔬 设计推导
    为什么把所有状态集中在一个地方？因为无状态组件可以随时崩溃和重启，只需从 GCS 恢复状态即可。这极大简化了容错——GCS 是唯一需要高可用的组件，其他都"可丢弃"。

    高可用方案：生产可配置外部 Redis 集群作为 GCS 后端存储。Head Node 故障时，新 Head Node 从 Redis 恢复状态。
    🏠 3.2 Raylet · 节点的管家
    每个节点（包括 Head Node）都跑一个 Raylet 守护进程，身兼三职：
    ①
    本地调度器：接收本节点 Worker 提交的任务，检查本地资源是否满足。满足→分配 Worker；不满足→转发给 GCS。
    ②
    资源管理器：跟踪本节点的 CPU、GPU、内存及自定义资源使用情况，定期向 GCS 汇报。
    ③
    对象管理器：管本节点 Plasma —— 本地对象创建/读取/驱逐 + 跨节点 Pull/Push + 对象溢出（Spilling，内存满时写磁盘）。
    📦 3.3 Plasma · 零拷贝数据层
    Plasma 是基于 Apache Arrow 的共享内存对象存储，每个节点一个实例。
    Plasma 数据流转的三种路径 
    # 同节点：零拷贝读
Worker A 创建对象 X → 写入 Plasma（mmap 区）
Worker B（同节点）  → mmap 映射，直接读，零拷贝

# 跨节点：网络拉取一次
Worker C（远程节点）→ Object Manager 检测到对象在远程
                  → 通过 gRPC 从源节点 Plasma 拉取
                  → 写入本地 Plasma → mmap 读

# 小对象（< 100KB）：直接 inline
返回值 5KB 的 dict → 不进 Plasma，直接打包在 RPC Response 中
    🔬 不可变性 · 关键约束
    Plasma 中的对象一旦创建就不可修改。这个设计消除了并发一致性问题——多个 Worker 可以同时读取同一对象，无需锁。代价是更新对象 = 创建新对象 + 旧对象等 GC（频繁更新场景请用 Actor）。
    ⚙️ 3.4 Worker Process · 计算执行器
    Worker 是实际执行计算的进程，分三种类型：
    类型生命周期特点
    Idle Worker等待分配预热进程池，减少启动开销
    Task Worker单次任务执行完归还进程池
    Actor Worker绑定 Actor生命周期跟随 Actor 实例，维护内部状态
    🌐 多语言支持
    Ray 通过 Core Worker（C++ 库）统一支持 Python、Java、C++ 前端。Core Worker 实现了所有权表（Ownership Table）、进程内存储和 gRPC 通信——这是 Ray 跨语言的基石。
    🎬 3.5 案例动画 · OCR + YOLO 在四层架构上是怎么跑的
    前面讲的 GCS / Raylet / Plasma / Worker 是静态拓扑。客户最关心的是动态过程： 一张合同扫描图进来，OCR 识别文字 + YOLO 检测印章/手写签名，结果写到 Lance——这个流程在四层组件上哪一步在哪？ 下面这张动画，把单张图的存储 + 计算 + 落盘从左到右串完一遍。
    ▶ 动画循环：粒子流展示了一张图的5 步流转——① S3 进 Plasma · ② OCR + YOLO 并行 mmap 读取 · ③ 各自写回本地 Plasma · ④ CPU Task 合并 · ⑤ 并行写 Lance。GCS 在顶部记账，Raylet 在底部调度，Plasma 在数据层流转，Worker 在计算层干活——四层各司其职。
    🔬 这张动画讲清了三件事
    1. 数据从不"汇聚到中心"：OCR 结果留在 GPU 1 的 Plasma，YOLO 结果留在 GPU 2 的 Plasma，CPU Task 来"取"，最后并行写 Lance——没有 Driver 单点瓶颈。
    2. OCR 与 YOLO 是真正并行的：同一个 Block_A 通过 mmap 在两个 GPU 节点同时被读取（Ray 通过 Object Manager Pull 一次到本地，之后零拷贝），双模型不抢同一张卡。
    3. 四层组件各司其职：GCS 记位置，Raylet 调度，Plasma 存数据，Worker 干活——任何一层都可以独立替换实现（比如 Plasma 换成 GPU Direct、Raylet 换成 K8s 调度器）而不影响其他层。这就是分层架构的"可演进性"。
    📌 售前一句话 在客户白板上一句话讲完拓扑：GCS 记账，Raylet 调度，Plasma 传数据，Worker 干活——组件越无状态越好恢复。客户 100 万图项目的部署：1 Head + 2 GPU + 4 CPU = 7 台节点起步。

    CHAPTER 04
    编程模型 · Task 与 Actor 双引擎
    OCR/YOLO 模型用 Actor，文本清洗用 Task
    第 4 幕 · 写代码前的关键拍板
    客户算法工程师急着开干："装饰器加 @ray.remote 不就完了？" 售前架构师叫停：错一个字，14 天变 8 分钟还是 14 小时，全看这一步。 PaddleOCR 模型加载要 5 秒、YOLO 模型加载要 3 秒——如果用 Task 写，每张图都重新加载模型，100 万张 = 5,000,000 秒模型加载。 必须用 Actor：模型常驻 GPU 显存，加载一次，后续推理 50ms/张。 而后处理（去噪/格式化）轻量、无状态，用 Task 弹性扩展。
    💡 核心区别
    Task = 远程函数（一次性，无状态，幂等）
    Actor = 远程对象（常驻，有状态，单线程）

    AI 工作负载这两个都需要：模型推理用 Actor（模型常驻显存），数据预处理用 Task（弹性扩展）。
    ⚡ 4.1 Task · 无状态的远程函数
    每次调用独立、异步返回 ObjectRef、可组合成 DAG、必须幂等（这是容错重建的前提）。
    Task 示例 · 声明资源需求

def train_step(data, model_state):

result = ray.get(ref)              # 真正等待执行结果

    无状态每次调用独立，不保留上下文
    异步.remote(...) 立即返回 ObjectRef
    可组合ObjectRef 可作为参数传给下一个 Task，自动构建 DAG
    幂等同一输入产生同一输出 —— 容错重建的前提
    🎭 4.2 Actor · 有状态的远程对象
    内部状态跨方法调用保持、单线程顺序执行（天然线程安全）、专属 Worker 不被复用。
    Actor 示例 · 模型常驻显存 
    @ray.remote(num_gpus=4)
class ModelServer:
    def __init__(self):
        self.model = load_model()  # 状态在 Actor 内持久化

    def predict(self, input_data):
        return self.model(input_data)

    def update_model(self, new_weights):
        self.model.load_weights(new_weights)

server = ModelServer.remote()              # 创建 Actor 实例
ref = server.predict.remote(image_batch)   # 调用方法，模型已在显存
    🔬 推导：为什么 Ray 需要 Actor？
    AI 场景大量存在有状态需求——模型参数、仿真环境状态、数据库连接池、向量索引。
    如果用无状态 Task 模拟有状态，每次都要序列化/反序列化几 GB 的模型权重，代价太高。
    Actor 让"状态"成为一等公民——这是 Ray 区别于 Spark 最关键的设计之一。
    同样跑 100 万张图：Task 每次 5s 加载模型，Actor 一次加载常驻——这是 OCR / 推理场景的关键
    📌 售前一句话 客户项目的拍板规则：OCR/YOLO 模型 → Actor（一次加载常驻显存）；文本清洗/检测框格式化 → Task（轻量弹性）。这一行决策，决定了项目是 8 分钟还是 14 小时。

    CHAPTER 05
    调度机制 · 自底向上的两级调度
    客户疑问：8 张 GPU 同时被抢，谁先谁后？
    第 5 幕 · 客户运维问到关键
    客户运维盯着监控图发问："100 万图同时排队，8 张 GPU 怎么分？分错一个就阻塞全局。" 售前架构师答：Ray 不是 Spark 那种"中心调度官"——而是 本地 Raylet 先抢答（~100μs）， 本地搞不定再上报 GCS 全局调度（~5ms）。 99% 的 OCR 任务在本地节点就分配完了，根本不堵全局。再叠加数据本地性： 图片在哪个节点的 Plasma，OCR Task 就优先调度到那个节点——10GB 图片省下跨节点传输。
    🚦 5.1 完整调度流程（7 步）
    Ray 的调度是整个系统最精妙的设计之一。我们从一次完整的任务调度来推导：
    ①
    Driver 调用 f.remote(args)，Core Worker 创建 TaskSpecification：函数、参数、资源需求
    ②
    NormalTaskSubmitter 向本地 Raylet 发送 RequestWorkerLease RPC
    ③
    本地 Raylet 检查资源：
      ✅ 资源足够 → 从 Worker 池分配 → 直接执行（快速路径，~100μs）
      ❌ 资源不足 → 转发给 GCS 全局调度
    ④
    GCS 查询全局资源视图，选择最优节点，返回候选 Raylet 地址
    ⑤
    本地 Raylet 将任务转发给目标 Raylet
    ⑥
    目标 Raylet 分配 Worker，Worker 从 GCS 获取函数定义、反序列化参数、执行
    ⑦
    结果存储：小对象内联返回（<100KB），大对象写入 Plasma
    两级调度：本地 Raylet 快速决策（~100μs）→ 不行才上 GCS 全局调度（~5ms）
    📍 5.2 数据本地性优化（Locality-Aware Scheduling）
    Ray 调度的核心原则：让计算靠近数据，而不是让数据靠近计算。
    反例 vs Ray 优化策略 
    big_data_ref = ray.put(huge_dataset)  # 10GB 数据在 Node A

@ray.remote
def process(data):
    return data.sum()

# 提交 100 个任务
refs = [process.remote(big_data_ref) for _ in range(100)]

# 朴素策略：100 个任务分散到各节点 → 每个节点都从 Node A 拉 10GB → 网络雪崩
# Ray 实际：调度器优先把任务分到 Node A（数据所在地）→ 0 跨节点传输
    ⚖️ 5.3 全局调度的负载均衡
    当本地无法调度时，GCS 进行全局调度：
    1
    可行性筛选：排除不满足硬性资源需求的节点
    2
    数据本地性评分：优先选择已有输入数据的节点
    3
    负载均衡：在候选节点中，从 top-k 最低负载节点中随机选择
    🔬 为什么是"top-k 随机"？
    如果总是选"最空闲"的节点，所有任务会涌向同一节点（惊群效应）。
    top-k 随机化打散了选择，避免了局部过载。
    📌 售前一句话 本地优先 + 数据本地性 + top-k 随机 —— 三招让 100 万张图调度延迟保持毫秒级，运维监控里"全局调度队列"永远是一条贴底线的平直曲线。

    CHAPTER 06
    容错机制 · 从血缘重建到 Actor 检查点
    客户最痛点：跑到 80% 时 GPU 节点炸了
    第 6 幕 · 凌晨 3 点的报警
    客户最深的恐惧："上次我们 Spark 作业跑到 13 天 23 小时崩了，从头再来。" 售前架构师必须把容错说清——Ray 的 3 道防线： ① ObjectRef 的 Owner 模型：每张图都有"户主"记录在案； ② Lineage Reconstruction（血缘重建）：GPU 节点 2 炸了？OCR 输出丢了？Ray 自动从源图重新跑一次 OCR； ③ Actor max_restarts + 检查点：OCRActor 进程崩了，GCS 在另一台 GPU 上重建 Actor，从最近检查点恢复。 客户跑到 80% 时单节点死掉，损失 = 那一个 Block 的几秒，不是 11 小时。
    👤 6.1 对象的 Owner 模型（Ray 2.0 引入）
    每个 ObjectRef 都有一个 Owner（创建该引用的 Worker）。Owner 负责：
    • 跟踪对象的位置
    • 在对象丢失时触发重建
    • 管理对象的生命周期（引用计数）
    Owner 关系示例 
    x = ray.put(1)         # Driver 拥有 x
y = f.remote(x)        # Driver 拥有 y（即使 y 在远程 Worker 上计算）

# Owner 死了 → ObjectRef 也就失效了
# 这就是为什么 Driver 一旦退出，所有引用都被回收
    🧬 6.2 Lineage Reconstruction · 血缘重建
    当节点故障导致对象丢失时，Ray 的处理流程：
    1
    先检查是否有副本在其他节点
    2
    没有副本 → 查找创建该对象的 Task
    3
    重新执行该 Task，递归重建其依赖
    4
    一直追溯，直到所有依赖都可用
    如果 Obj Y 也丢了，继续追溯重跑 Task B；血缘是一棵 DAG
    ⚠️ 限制
    • ray.put() 创建的对象不可重建（不知道如何重新生成）
    • Actor 任务创建的对象默认不可重建（Actor 有状态，不保证幂等）
    • Task 必须是确定性的，否则重建结果不一致
    💾 6.3 Actor 容错
    Actor 容错分两层：
    无状态 Actor 方法：和普通 Task 一样通过血缘重建
    有状态 Actor：需要用户实现 检查点（Checkpoint）
    Actor 检查点模式 
    @ray.remote(max_restarts=3, max_task_retries=3)
class StatefulActor:
    def __init__(self):
        self.state = self._load_checkpoint() or init_state()

    def update(self, delta):
        self.state += delta
        self._save_checkpoint(self.state)  # 用户主动保存检查点
        return self.state
    📌 售前一句话 给客户的承诺：OCR Task 失败 Ray 自动重做（血缘），OCRActor 节点死了 GCS 自动重建（max_restarts + 检查点）。100 万图跑到 80% 死一台 GPU = 损失几秒，不是 11 小时。

    CHAPTER 07
    高级调度 · Placement Group 与 Gang Scheduling
    客户新需求：OCR 和 YOLO 必须同节点协同
    第 7 幕 · 客户加码：双模型必须同生共死
    客户算法经理临门一脚："我们要 OCR 和 YOLO 共享一张图的中间结果——OCR 切出来的文字区域要喂给 YOLO 检测印章和签名位置。" 这意味着同一张图的 OCRActor 和 YOLOActor 必须在同一节点，否则中间 Tensor 跨节点拉，网络全部打满。 售前架构师上 Placement Group + STRICT_PACK 策略： "要么这两个 Actor 一起在同一台机器启动，要么一个都不启动。" 这就是分布式训练界的"Gang Scheduling"——也是 Ray 比 Spark 更适配 AI 工作负载的关键证据。
    ⚠️ 真实痛点
    分布式训练有一个硬性要求：所有 Worker 必须同时启动，否则 AllReduce 卡死，训练无法开始。
    如果用普通调度，可能 3 个 Worker 拿到 GPU 启动了，第 4 个还在排队 → 浪费且死锁。
    这就是 Gang Scheduling（全有或全无调度）必须解决的问题。
    📦 7.1 Placement Group · 原子性资源预留
    Ray 的 Placement Group 提供 原子性的 bundle 预留——要么所有 bundle 都成功，要么全部失败。
    Placement Group 完整示例 
    from ray.util.placement_group import placement_group
from ray.util.scheduling_strategies import PlacementGroupSchedulingStrategy

# 预留 4 个 bundle，每个需要 1 CPU + 1 GPU
pg = placement_group([{"CPU": 1, "GPU": 1}] * 4, strategy="STRICT_PACK")
ray.get(pg.ready())  # 等待所有 bundle 预留成功（原子性）

@ray.remote(num_gpus=1)
def train_worker(rank):
    return f"Worker {rank} training"

# 在预留资源上调度，bundle_index 指定每个 worker 用哪个 bundle
results = ray.get([
    train_worker.options(
        scheduling_strategy=PlacementGroupSchedulingStrategy(
            placement_group=pg, bundle_index=i)
    ).remote(i)
    for i in range(4)
])
    🎯 7.2 四种放置策略
    Ray 提供 4 种策略，覆盖从"极致打包"到"极致分散"的所有需求：
    策略行为适用场景
    STRICT_PACK所有 bundle 必须在同一节点需要 NVLink 的高速 GPU 互联
    PACK尽量打包，放不下则分散优先数据局部性，允许降级
    STRICT_SPREAD每个 bundle 必须在不同节点最大化容错
    SPREAD尽量分散，不够则共享负载均衡，允许降级
    四种策略适配从"AllReduce 训练（PACK）"到"高可用部署（SPREAD）"的全光谱场景
    📌 售前一句话 客户的双模型协同方案：每张 GPU 卡建一个 STRICT_PACK PG，预留 OCR + YOLO 各 0.5 GPU——两个 Actor 同节点共享显存与 CUDA 上下文，中间 Tensor 走 mmap 零拷贝。这是 Spark 永远做不到的事。

    1
    为每个 Block 创建一个远程 Task，资源声明 num_gpus=1
    2

    3
    本节点 GPU 满载 → GCS 全局调度器查找有空闲 GPU 的远程节点
    4

    🎭 8.3 Phase 4 · 计算：为什么用 Actor 而不是 Task？
    这是性能能否成立的关键决策。
    方案 A · 朴素 Task（每次加载模型）—— 错误示范 
    @ray.remote(num_gpus=1)
def ocr_task(image_batch):
    model = PaddleOCR()    # 每次 Task 都要加载模型 ~5 秒！
    return model.ocr(image_batch)

# 1000 个 Block × 5 秒 = 8333 秒（2.3 小时）浪费在加载上 ❌
    方案 B · Actor Pool（模型常驻）—— 正确做法 
    @ray.remote(num_gpus=1)
class OCRActor:
    def __init__(self):
        self.model = PaddleOCR()  # 启动时加载一次，常驻 GPU 显存

    def recognize(self, image_batch):
        return self.model.ocr(image_batch)  # 后续直接推理 ~50ms

from ray.util.actor_pool import ActorPool
actors = [OCRActor.remote() for _ in range(num_gpus)]
pool = ActorPool(actors)

for batch in image_batches:
    pool.submit(lambda a, v: a.recognize.remote(v), batch)
results = list(pool.get_next_unordered())

    ①
    第一跳 · S3 → Driver → Plasma（数据进入 Ray 世界）
    S3: /images/doc_00421.png (312 KB)
  ↓ ray.data.read_images() 内部执行 ray.put()
Driver Core Worker:
  1. CloudPickle 序列化 → Arrow RecordBatch
  2. 写入 Head Node Plasma（mmap 区）
  3. 生成 ObjectRef(ID=0x7fa3...01, owner=Driver)
  4. ObjectRef 注册到 Driver Ownership Table
  5. Driver 向 GCS 报告：对象位于 Head Node
    ⚠️ read_images() 是惰性的——先构建 Block 元数据（文件列表+偏移），实际读取延迟到 Task 执行时。.repartition() 和 .map_batches() 也只是构建执行计划，不触发数据加载。
    ②
    第二跳 · Driver → Raylet → GPU Worker（调度 + 数据本地性）
    Driver 调用 .map_batches(ocr_fn) 触发实际执行:
  1. 构造 TaskSpecification:
     - function: ocr_fn (从 GCS 拉序列化函数体)
     - args: [ObjectRef(Block_A)] ← 按引用传递
     - resources: num_gpus=1
  2. Driver → 本地 Raylet: RequestWorkerLease
  3. 本地 Raylet 检查:
     A. Head Node 有空闲 GPU → 本地分配（快速路径 ~100μs）
     B. Head Node GPU 满载 → 回复"去问 GCS"
  4. GCS 全局调度器决策:
     - 查 Block_A 位置 → Head Node
     - 查 GPU 资源 → GPU Node 1 有空余
     - 决策: 牺牲数据本地性，调到 GPU Node 1
  5. 转发到 GPU Node 1 Raylet → 分配 Worker
    🌐 唯一一次跨节点传输：GPU Node 1 的 Object Manager 通过 gRPC 从 Head Node Plasma 拉取 Block_A，写入本地 Plasma。后续同节点读全是 mmap。
    ③
    第三跳 · Plasma → Actor Worker → GPU 推理（零拷贝执行）
    GPU Node 1 上 Task Worker 被分配后:
  1. Core Worker (C++) 接收 PushTask RPC
  2. 解析参数：args[0] 是 ObjectRef(Block_A)
     - 检查本地 Plasma：Block_A 已在（上一步拉的）
     - mmap 映射到 Worker 进程地址空间 → 零拷贝
  3. 从 GCS 拉 ocr_fn 序列化函数体，反序列化
  4. 走 Actor Pool 模式：
     Task Worker → OCR Actor.recognize.remote(batch)
  5. OCR Actor Worker（GPU Node 1 常驻）:
     - mmap 读 image_batch from Plasma
     - PaddleOCR 推理: bytes → GPU Tensor → 文本
     - 推理 ~50ms (batch=32 时 ~1.6s)
    💡 为什么不是 Task 直接调 OCR？因为 Task Worker 一次性，模型权重随之释放。Actor Worker 跟随 Actor 实例，模型常驻 GPU 显存。
    ④
    第四跳 · GPU Worker → Plasma → ObjectRef（结果回写）
    OCR Actor 执行完毕，Core Worker 处理返回值:

  情况 A: 返回值 < 100KB（单张图 OCR 文本）
  → 直接 inline 在 PushTask Response 中
  → 不经过 Plasma，延迟最低
  → ObjectRef 标记为 "inlined"

  情况 B: 返回值 ≥ 100KB（batch=32 一批结果）
  → 写入本地 Plasma
  → ObjectRef 指向 Plasma 位置
  → Driver Ownership Table 记录: 对象位于 GPU Node 1
  → Raylet 心跳同步对象位置到 GCS
    ⑤
    第五跳 · ObjectRef(OCR结果) → CPU Worker（跨节点后处理）
    后处理 Task: postprocess(ObjectRef(OCR_Result))
  resources: num_cpus=0.5

调度流程:
  1. 本地 Raylet (GPU Node 1) 检查: GPU 节点 CPU 配额少，无空余
  2. GCS 全局调度: 选 CPU Node N（CPU 充足）
  3. CPU Node N Object Manager 检测到 OCR_Result 在远程
  4. 从 GPU Node 1 Plasma gRPC Pull → CPU Node N Plasma
  5. CPU Worker mmap 读 → 反序列化 → clean_text → format
  6. 结果写回 CPU Node N Plasma（或 inline 返回）
    📊 总跨节点传输次数：图片原始数据 1 次（Hop2）+ OCR 结果 1 次（Hop5）= 每段数据网络中只传 1 次，同节点全 mmap。

    第六跳 · ray.get() → Write（最终收集与持久化）

    Autoscaler 跑在 Head Node，定期扫 GCS 的资源视图：
    Autoscaler 决策时序 
    场景：100 万张图任务刚启动，1000 个 Block Task 排队中

  1. Autoscaler 发现: pending=800, available_GPUs=4, needed≈32
  2. 向云平台申请 28 个 GPU 节点（Ray Worker 形式加入集群）
  3. 新节点启动 → raylet 注册 GCS → 调度器分配排队 Task
  4. 任务陆续完成 → pending 下降

GPU 节点变化: 4 → 32 → 16 → 4（按队列深度伸缩）
CPU 节点变化: 2 → 8 → 4 → 2（后处理量小，缩得更快）

    🔬 11.4 架构设计的深层思考
    看到这里，你已经掌握了 Ray 的全部架构。但还有三个设计权衡值得反思：
    ⚖️ 思考一 · GCS 单点问题
    GCS 是 Ray 的阿喀琉斯之踵。所有其他组件无状态化简化了容错，但 GCS 本身成为单点。缓解策略：
    • GCS 不参与数据通路，只管元数据，吞吐压力有限
    • 支持外部 Redis 集群作持久化后端（生产标配）
    • GCS Flushing：定期将内存中的血缘信息刷盘，避免无限增长
    ⚖️ 思考二 · 不可变对象的权衡
    Plasma 不可变设计消除了并发问题，但也意味着：
    • 更新一个对象 = 创建新对象 + 旧对象等待 GC
    • 对于频繁更新的场景（模型参数热更新），Actor 模式是更好的选择
    • ObjectRef 的引用计数和 GC 是分布式系统中的难题——这就是 Owner 模型如此重要的原因
    ⚖️ 思考三 · 从 Kubernetes 到 Ray 的范式迁移
    K8s 管理的是 容器（粗粒度、长生命周期），Ray 管理的是 任务（细粒度、短生命周期）。
    Ray Autoscaler 可以根据任务队列深度动态增减节点，这是传统 K8s HPA 做不到的。

    在 KubeRay 加持下，Ray 跑在 K8s 之上，形成 "K8s 管理节点/存储 + Ray 管理计算任务" 的分层架构——这也是腾讯云 Ray（EMR on TKE）的技术路线。
    📌 售前一句话 给客户工程师的临别赠言：Ray 的"简单"背后是深思熟虑的架构权衡——GCS 单点换其他组件无状态化；对象不可变换零拷贝；自底向上调度换低延迟。这三个 trade-off 想清楚，三个月后客户自己就能调优了。
