CHAPTER 00
为什么我们需要 Ray？
从一台机器算不动开始
⚠️ 真实场景
你有 100 GB 的图片要过 CLIP 模型抽特征。
一张 V100 GPU 单卡跑：约 12 小时。截止时间是明天早 9 点。怎么办？
💡 直觉
"把活分给一群机器一起干"——这就是分布式计算。10 张 GPU 并行 → 1.2 小时 ✓

但写过 MPI / Spark / 自己 socket 通信的人都知道：
分布式编程的痛点 90% 不在算法，而在把单机代码改成分布式。
把数据切成 10 份，10 张 GPU 同时算
🚀 Ray 的承诺
"让分布式像写本地 Python 一样简单"——给函数加一行 @ray.remote， 它就能在集群任意机器跑。Ray 帮你处理：调度、通信、容错、资源、序列化。
"分布式 Hello World" 
import ray
ray.init()  # 启动/连接集群

@ray.remote  # ← 唯一改动：加这一行
def slow_square(x):
    import time; time.sleep(1)
    return x * x

# 单机串行: 10 秒；分布式: ~1 秒（10 个 task 并行）
futures = [slow_square.remote(i) for i in range(10)]
results = ray.get(futures)  # [0, 1, 4, 9, ..., 81]
📌 一句话 Ray = 用本地 Python 语法表达分布式计算。后面 8 章拆解 Ray 怎么做到。
下一章：集群是谁组成的？↓
CHAPTER 01
集群里的三种角色
谁指挥？谁干活？谁记账？
💡 类比 · 一家餐厅
🧑‍🍳 老板（Driver）：接订单、决定做什么菜、把单子贴到票墙
👨‍🍳 厨师（Worker）：盯票墙，抢单做菜
📋 票墙（GCS）：所有订单的真理，所有人来这里查"现在该干啥"
绿色心跳：worker 周期上报存活；金色：driver 提交 task
📖 三个角色
  • Driver：你的 python my_script.py，入口进程。发起 task 调用，但不做实际计算。
  • Worker：每台机器上的"工人进程"，跑你的 @ray.remote 函数（默认每核一个）。
  • GCS：跑在 head node 上的元数据中心。所有节点心跳上报，所有 task/actor 状态登记在这。挂了集群就瘫——生产要做高可用（Redis backup）。
启动一个 4 节点集群 
# head node（这台跑 GCS）
ray start --head --port=6379

# 其他 3 台 worker node 连过来
ray start --address='10.0.0.1:6379'

# Driver 端：
import ray
ray.init(address='auto')
print(ray.cluster_resources())
# {'CPU': 32.0, 'GPU': 4.0, 'memory': 128_000_000_000, ...}
📌 一句话 集群 = 1 个 Driver + N 个 Worker + 1 个 GCS。
下一章：Driver 怎么把活派出去？↓
CHAPTER 02
Task · 无状态函数调用
把一个函数调用扔到任意机器去算
💡 直觉
本地 y = f(x) 是同步阻塞——程序停在这一行等结果。
Ray 把它改成 y = f.remote(x) → 立刻返回 ObjectRef（凭证），任务在别的机器后台跑。 等你需要值时 ray.get(y) 才阻塞拿结果。

所以你可以一口气派出 1000 个 task，1000 台 worker 同时算。
Driver 一次性派 3 个 task，scheduler 分发给 3 个 worker 并行算
🔑 关键 API
  • @ray.remote → 把普通函数变成"远程函数"
  • f.remote(x) → 异步调用，立返 ObjectRef
  • ray.get(ref) → 阻塞取值（可单个或 list）
  • ray.wait([refs]) → "谁先好返回谁"，做流水线必备
对比：串行 vs 并行 
import ray, time
ray.init()

@ray.remote
def crawl(url):
    time.sleep(1)  # 模拟网络 IO
    return len(url)

urls = [f"https://site-{i}" for i in range(100)]
# 串行：100 秒；并行：~1 秒
futures = [crawl.remote(u) for u in urls]
results = ray.get(futures)
📌 一句话 Task = 一次性的远程函数调用，无状态，跑完即销毁。适合纯计算。
下一章：但模型加载很贵，每次都重新加载怎么办？↓
CHAPTER 03
Actor · 有状态服务
让模型只加载一次
⚠️ Task 的局限
CLIP 模型加载到 GPU 要 5 秒。如果每个 task 都加载一次： 100 张图 × 5 秒 = 500 秒，全花在加载上。实际推理一张图只要 0.05 秒。
💡 直觉
把 worker 进程"固定下来"：开一次就常驻，模型在内存里不卸载。 后续所有调用都复用这个 worker 的内存——这就是 Actor。

在 Python 里这正好就是对象方法调用：actor.method(x)。
Actor = 能远程调用方法的、有状态的、长寿的对象。
小球 = 推理请求，被分发到 4 个常驻 Actor，避免重复加载模型
🎭 Task vs Actor
维度Task（@ray.remote 函数）Actor（@ray.remote 类）
状态无状态有状态（实例属性）
生命周期跑完即销毁常驻直到 ray.kill()
调用f.remote(x)actor.method.remote(x)
用途纯计算 / 一次性 IO模型推理 / 缓存 / 状态机
资源每次重新分配一次分配，长期占用

下一章：多个 task 都要用同一份大数据怎么办？↓
CHAPTER 04
Object Store · 跨进程共享内存
大对象只存一份，零拷贝读
⚠️ 痛点
你有 2 GB 的训练矩阵 W，要喂给 100 个 task：
[f.remote(W, x_i) for i in range(100)]
如果每个 task 都拷贝一遍 → 200 GB 网络传输，集群带宽爆炸。
💡 直觉
把 W 放进每个节点上的共享内存（Plasma），所有本机 worker 直接 memory map 读， 不复制不拷贝——这叫 zero-copy。
跨节点访问时，Ray 自动把对象在节点间复制一次，缓存下来供后续 task 复用。
Plasma = 每节点本地共享内存；同节点 zero-copy，跨节点首次复制 + 缓存
🧠 关键事实
  • 对象不可变：放进 store 后不能改（要改先生成新对象）。让缓存/共享/容错都简单。
  • 引用计数自动 GC：所有 ObjectRef 都销毁后，对象自动从 store 删除。
  • numpy 数组特别快：直接 memory map，无需 pickle 序列化。
  • 会 spill 到磁盘：内存满时 Ray 自动把冷对象 spill 到 /tmp/ray，不至于 OOM。
显式 ray.put / 隐式传递的区别 
import ray, numpy as np
ray.init()
W = np.random.randn(10000, 10000).astype('float32')  # 400 MB

@ray.remote
def use(W, x): return W @ x

# ❌ 反例：每次 remote 都把 W 序列化进 store 一份（100 份）
results = [use.remote(W, x_i) for i in range(100)]

# ✓ 正解：先 ray.put 一次，再以 ref 传递
W_ref = ray.put(W)  # 进 Plasma 一次
results = [use.remote(W_ref, x_i) for i in range(100)]
📌 一句话 Object Store = 分布式版本的"大对象池"。原则：大对象先 ray.put 拿 ref，再以 ref 传递。
下一章：调度器是怎么决定 task 跑哪台机器的？↓
CHAPTER 05
调度器 · 任务往哪儿跑？
资源约束 + 数据本地化
💡 调度的两条朴素原则
① 资源够不够：GPU task 只能派给有 GPU 空闲的 worker，CPU task 找 CPU 槽位。
② 数据离得近不近：如果 task 要读的对象 W 在 node-2 的 Plasma 里， 优先派给 node-2 的 worker → 省一次跨节点拷贝（locality-aware scheduling）。
A 要 GPU → node-2；B 数据在 node-1；C 引用对象 W 在 node-3
🛠 资源声明
@ray.remote(num_cpus=2, num_gpus=1, memory=4_000_000_000)
def train_step(batch): ...

# 自定义资源（按机器类型 / 标签）
@ray.remote(resources={"high_mem": 1})
def huge_join(df): ...
📌 一句话 调度器目标：把每个 task 派到既满足资源、又离数据近的 worker，最大化吞吐 + 最小化网络。
下一章：节点挂了怎么办？↓
CHAPTER 06
容错 · 节点挂了怎么办
lineage 重算 + 自动 retry
⚠️ 真实故障
训练任务跑了 5 小时，集群里的 node-7 网卡抽风掉线。 node-7 上跑的 12 个 task + 缓存的中间对象全没了。5 小时白费？
💡 直觉 · lineage（血缘）
Ray 记得每个对象是怎么算出来的：obj_X = f(obj_A, obj_B)。
如果 obj_X 因为节点宕机丢了，Ray 顺着血缘：找到 f 和它的输入 → 在另一台机器重新执行 → 得到新的 obj_X。
整个集群就是个可重放的 DAG。
输入对象 A、B 还在（其他节点 Plasma 缓存）→ 只重算 X、Y
🛡 三层容错
  • Task 失败：默认自动 retry 3 次，可 max_retries=N 调整。
  • Worker 进程崩：所属 task 重新创建在其他节点。
  • 整个 Node 宕机：丢失的对象按 lineage 重算；Actor 默认不重启（max_restarts=N 开启）。
  • GCS 高可用：开启 Redis backup 后，head node 故障也能恢复（生产标配）。
容错 API 
@ray.remote(max_retries=3)  # task 自动重试 3 次
def flaky_io(url): ...

@ray.remote(max_restarts=5, max_task_retries=2)
class StatefulCache:
    def __init__(self): self.cache = {}

# 启动 head 时开启 GCS HA
# RAY_REDIS_ADDRESS=<redis_host>:6379 ray start --head ...
📌 一句话 Ray 容错 = 整个集群是可重放的 DAG，丢什么算什么，不必整体重跑。
下一章：100GB 数据装不进内存怎么处理？↓
CHAPTER 07
Ray Data · 流式数据流水线
100GB 不能一次 load？切成 block 流过去
⚠️ 痛点
100 GB 图片要做：读 → 解码 → resize → CLIP 推理 → 写 parquet。 如果一次性 load 进内存：OOM。如果一步一步落盘再读：磁盘 IO 翻倍。
💡 直觉 · 流水线 + block 级并行
把数据切成 block（一块通常 ~128MB / 1k 行）。 每个 block 独立流过整条 pipeline，前一个 block 在做 CLIP 推理时，下一个 block 已经在解码—— 像工厂流水线一样，每一阶段都同时有 block 在加工，吞吐 = 最慢一段的 1/throughput。
同一时刻每个 stage 都在处理不同的 block，CPU/GPU 全程满载
🌊 Ray Data 关键概念
  • Dataset：分布式数据集，由 N 个 block 组成。
  • Block：基本调度单位（默认 ~128MB），底层是 Arrow / pandas DataFrame。
  • 惰性执行：map / filter 不立刻跑，show() / iter_batches() / write_*() 才触发。
  • 流式 pipeline：所有 stage 同时跑，前一阶段的 block 一就绪就喂下一阶段。
  • Backpressure：下游慢时上游自动减速，防止内存爆炸。
实战：图片库 → CLIP 特征 → parquet 
import ray
ds = ray.data.read_images("s3://my-bucket/photos/", size=(224, 224))

# 关键：用 actor 池跑 GPU 推理（避免每个 task 重新加载模型）
class CLIPInfer:
    def __init__(self):
        import clip; self.model, _ = clip.load("ViT-B/32", device="cuda")
    def __call__(self, batch):
        batch["feat"] = self.model.encode_image(batch["image"]).cpu().numpy()
        return batch

ds = ds.map_batches(
    CLIPInfer,
    concurrency=4,           # 4 个 actor
    num_gpus=1,               # 每个 actor 1 GPU
    batch_size=32,            # 充分利用 GPU 显存
)

ds.write_parquet("s3://my-bucket/features/")  # 触发执行
📌 一句话 Ray Data = 分布式版本的 pandas/Spark Dataset，但天生支持 GPU + 流式 + Python 任意代码。AI 数据处理的"事实标准"。

CHAPTER 08
Ray 生态全景
5 个高层库，覆盖 ML 全链路
💡 一张图记住
最底层是 Ray Core（task / actor / object）。
往上是5 个面向场景的高层库，全都用 Ray Core 实现，所以彼此共享集群、共享对象、无缝拼接。
这就是 Ray 的杀手锏：一个集群跑 ML 全链路，不用维护 5 套基础设施。
所有库共享同一集群和对象存储——上一步的输出可直接喂下一步
🔗 串成一条 ML pipeline
阶段用什么关键 API
① 数据准备Ray Dataray.data.read_images / map_batches
② 超参搜索Ray Tunetune.Tuner(trainable, search_space)
③ 分布式训练Ray TrainTorchTrainer(scaling_config=...)
④ 上线服务Ray Serve@serve.deployment / Deployment.bind()
⑤ 强化学习（可选）RLlibPPOConfig().build().train()

import ray
from ray import train, tune, serve
from ray.train.torch import TorchTrainer

# ① Ray Data: 准备数据集
ds = ray.data.read_parquet("s3://...features/")

# ② Ray Tune × Train: 调参 + 训练
def train_loop(config):
    model = build_model(hidden=config["hidden"])
    for batch in train.get_dataset_shard("train").iter_torch_batches():
        step(model, batch)

trainer = TorchTrainer(train_loop, datasets={"train": ds},
    scaling_config=train.ScalingConfig(num_workers=4, use_gpu=True))
tuner = tune.Tuner(trainer, param_space={"hidden": tune.choice([128,256])})
best = tuner.fit().get_best_result()

# ③ Ray Serve: 上线推理
@serve.deployment(num_replicas=3, ray_actor_options={"num_gpus": 1})
class Predictor:
    def __init__(self): self.model = load_from(best.checkpoint)
    async def __call__(self, req): return self.model(req.json())

serve.run(Predictor.bind())  # 启动 HTTP 服务
